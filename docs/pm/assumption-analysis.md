# Assumption Analysis — WordPang v3.2 기능

**Date:** 2026-03-22
**Framework:** 4가지 위험 영역 (Value / Usability / Viability / Feasibility)
**분석 기능:** v3.2 핵심 3개 기능 (Streak 회복, 복습 배지, 발음 오디오)

## Feature A: Streak 회복 유예 (1일 Grace Period)

**기능 요약:** streak 끊긴 다음 날 하루 내 플레이하면 이전 streak 복구

### Value 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **V1** ✅ | streak 끊김이 실제로 이탈 원인이다 | 높음 | OST 고객 인터뷰 + Duolingo 사례로 검증됨 |
| **V2** ⚠️ | "오늘 복구 가능!" 메시지가 재방문을 유발한다 | 중간 | 메시지를 봐야 효과 있음 — 앱을 열지 않으면 볼 수 없음 |
| **V3** ⚠️ | 아이(초등학생)가 streak 가치를 이해한다 | 중간 | 7~8세는 streak 개념보다 즉각 보상 선호 가능 |
| **V4** ❓ | 1일 유예가 너무 관대해서 streak 긴장감을 낮추지 않는다 | 낮음 | 2일 이상이면 streak 긴장감 완전 소멸 → 1일만 제공이 올바른 밸런스 |

**PM 시각**: Duolingo, Headspace 모두 streak freeze 제공 — 업계 검증된 패턴. 단, 학부모가 아이 대신 앱을 열어서 streak을 "구해주는" 시나리오 발생 가능 (의도치 않은 사용)

### Usability 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **U1** ✅ | 아이가 "오늘 복구 가능!" 메시지를 이해한다 | 높음 | 한국어 메시지 + 이모지 ⚡ → 직관적 |
| **U2** ⚠️ | 홈 화면에서 회복 메시지가 눈에 띈다 | 중간 | 디자인 prominence 중요 — 작으면 무시됨 |
| **U3** ⚠️ | streak 회복이 완료됐을 때 피드백이 명확하다 | 중간 | "🔥 Streak 복구!" 토스트 메시지로 확인 필요 |

**Designer 시각**: streak 회복 상태와 정상 streak을 시각적으로 구분해야 함. 회복 중 streak은 다른 색(예: 점선 불꽃)으로 표시 → 아이에게 "특별 상태" 인식

### Viability 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **Vi1** ✅ | streak 회복 로직이 기존 통계와 호환된다 | 높음 | stats.ts + storage.ts 구조 확인 완료 |
| **Vi2** ✅ | 오프라인 환경에서 날짜 계산이 정확하다 | 높음 | AsyncStorage에 streak 끊긴 날짜 저장 → 로컬 계산 |
| **Vi3** ⚠️ | 학부모가 streak 회복을 "부정행위"로 느끼지 않는다 | 중간 | 업계 표준 패턴이므로 대부분 수용하지만, 완벽주의 학부모는 거부 가능 |

### Feasibility 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **Fe1** ✅ | 기기 시간 조작으로 streak을 무한 복구할 수 없다 | 높음 | 서버 없음 → 기기 시간 조작 가능하지만, 초등학생 대상이므로 허용 수준 |
| **Fe2** ✅ | streak 끊긴 날짜 저장 로직이 4시간 내 구현 가능 | 높음 | AsyncStorage 키 추가 + 날짜 비교 로직 간단 |

**Engineer 시각**: `wordpang_streak_broken_date` 키 추가 → 앱 실행 시 오늘 날짜와 비교. 엣지 케이스: 기기 날짜 변경, 23:59에 게임 시작 → 명세서에 "같은 달력 날 기준"으로 정의 필요

### 최우선 검증 가정
- **V2** + **V3**: "아이가 앱을 열어야 메시지를 본다" → 복습 배지(F2)와 함께 OST E3 실험

## Feature B: 홈 화면 복습 배지 ("복습 N개 준비됨")

**기능 요약:** 복습 큐에 항목이 있으면 홈 화면에 카드 표시

### Value 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **V1** ✅ | 복습 큐에 단어가 실제로 쌓인다 | 높음 | 현재 spaced-repetition.ts 동작 확인 |
| **V2** ⚠️ | 아이가 "복습"의 가치를 인지한다 | 낮음 | 초등학생은 "복습"보다 "새 게임"을 선호할 수 있음 |
| **V3** ⚠️ | 복습 배지가 앱 재방문 트리거가 된다 | 중간 | 배지를 보려면 앱을 열어야 함 — 앱 아이콘 배지와 병행 필요? |

**PM 시각**: 배지 메시지를 "복습 N개" 대신 "지난번 단어 기억 테스트! 🧪" 등 게임 언어로 바꾸면 아이 관심 더 높아질 가능성

### Usability 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **U1** ✅ | "복습 N개 준비됨" 텍스트를 초등학생이 읽고 이해한다 | 높음 | 한국어 + 숫자로 명확 |
| **U2** ⚠️ | 카드를 탭하면 복습 화면으로 바로 간다는 것을 안다 | 중간 | "탭 = 이동" CTA 명확성 → 화살표 아이콘 또는 버튼 스타일 필요 |
| **U3** ✅ | 복습 완료 후 홈 돌아왔을 때 카드가 업데이트된다 | 높음 | AsyncStorage 읽기 + useFocusEffect 훅으로 처리 |

**Designer 시각**: SketchyBox 스타일 카드에 "📚 복습 7개 준비됨!" 형태. 복습 큐가 0이면 카드 사라짐 → 사용자가 잘 모를 수 있음, 애니메이션 fade-out 고려

### Viability 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **Vi1** ✅ | 복습 큐 데이터가 항상 정확하다 | 높음 | review-queue.ts + AsyncStorage 구조 안정적 |
| **Vi2** ⚠️ | 복습 카드가 홈 화면 레이아웃과 충돌하지 않는다 | 중간 | 홈 화면 상단 레이아웃 리뷰 필요 (기존 Streak/Stats 카드와 공간 경쟁) |

### Feasibility 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **Fe1** ✅ | 복습 큐 카운트 읽기 로직이 3시간 내 구현 가능 | 높음 | `getReviewQueue()` 함수 이미 존재 |
| **Fe2** ✅ | useFocusEffect로 화면 복귀 시 카운트 갱신 가능 | 높음 | React Navigation useFocusEffect 표준 패턴 |

**Engineer 시각**: 성능 최적화 포인트 — 홈 화면 마운트마다 AsyncStorage 읽기는 부담 없음 (데이터 소량). 단, 복습 큐 계산 알고리즘 복잡도 확인 필요 (현재 O(n) 선형)

### 최우선 검증 가정
- **V2**: "아이가 복습 가치를 인지한다" → 출시 후 복습 진입률 비교 (OST E2)

## Feature C: 발음 오디오 (학습 카드 스피커 버튼)

**기능 요약:** 학습 카드에서 정답 단어 발음을 TTS 또는 MP3로 재생

### Value 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **V1** ✅ | 발음 오디오가 학습 효과를 향상시킨다 | 높음 | 음성 기반 언어 학습 연구로 검증됨 |
| **V2** ⚠️ | 학부모가 발음 정확성을 기대한다 | 중간 | 한국 학부모는 발음에 민감 → TTS 품질이 낮으면 신뢰도 훼손 |
| **V3** ⚠️ | 아이가 스피커 버튼을 반복 탭한다 (학습 강화) | 중간 | 아이가 버튼을 장난으로 반복 탭할 가능성 → 디바운스 필요 |

**PM 시각**: 학부모 MOT #2 (유지 결정) 강화 — "아이가 발음도 배운다"는 가시적 교육 가치. App Store 리뷰에서 "발음 기능 있으면 더 좋겠다" 패턴이 나오면 강력한 수요 신호

### Usability 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **U1** ✅ | 스피커 아이콘(🔊)이 "소리 재생"임을 아이가 안다 | 높음 | 범용 UI 패턴, 7세도 인지 |
| **U2** ⚠️ | 오디오 재생 중 아이콘 애니메이션이 명확하다 | 중간 | 재생 중 시각 피드백 없으면 "버튼이 안 눌렸나?" 혼란 |
| **U3** ⚠️ | 무음 모드에서도 재생된다는 것을 사용자가 안다 | 낮음 | iOS 무음 모드 = 시스템 사운드 무음, 앱 내 오디오는 별도 — 학부모 교육 필요 |

**Designer 시각**: 재생 중 스피커 아이콘에 wave 애니메이션 추가 (Lottie 또는 CSS 애니메이션). 재생 완료 후 아이콘 reset. 힌트 패널에도 동일한 스피커 버튼 패턴 재사용.

### Viability 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **Vi1** ⚠️ | expo-speech TTS 음질이 학부모에게 수용 가능하다 | 낮음 | expo-speech는 iOS 내장 TTS — 품질은 보통 수준. 학부모 테스트 필요 |
| **Vi2** ⚠️ | 514개 단어 모두 TTS로 자연스럽게 발음된다 | 중간 | 고유명사, 의성어 등 TTS 오발음 단어 존재 가능 → 예외 처리 |
| **Vi3** ✅ | 오프라인 환경에서 오디오 재생된다 | 높음 | expo-speech는 인터넷 연결 불필요 (iOS 내장 TTS) |

### Feasibility 가정

| 가정 | 세부 내용 | 신뢰도 | 검증 방법 |
|------|---------|--------|---------|
| **Fe1** ✅ | expo-av/expo-speech 통합이 기존 사운드 시스템과 충돌하지 않는다 | 중간 | sound.ts (pop, flip, win, lose, shake)와 오디오 채널 분리 필요 — User Story S2 AC5 |
| **Fe2** ✅ | 514개 단어 발음이 별도 녹음 없이 TTS로 커버된다 | 높음 | 영어 TTS는 일반 단어 거의 모두 처리 가능 |
| **Fe3** ⚠️ | 오디오 재생이 기기 볼륨 설정에 따라 정상 작동한다 | 중간 | iOS 볼륨 카테고리 설정(AVAudioSession) 확인 필요 — 무음 모드 우회 |

**Engineer 시각**:
1. `Speech.speak(word, { language: 'en-US', rate: 0.8 })` — expo-speech로 간단 구현
2. 기존 `sound.ts`의 expo-av와 충돌 방지: AVAudioSession category `.playback` 설정
3. 디바운스 200ms 추가 (연속 탭 방지)
4. 재생 상태 `isPlaying: boolean` 관리 → 아이콘 애니메이션 연동

### 최우선 검증 가정
- **Vi1**: expo-speech TTS 음질 → TestFlight 학부모 5명에게 "이 발음이 자연스러운가?" 설문
- **Fe1**: 오디오 충돌 → 개발 환경에서 게임 중 효과음 + TTS 동시 재생 테스트

### 즉시 실행 검증 실험

1. **TTS 음질 테스트** (1일): expo-speech로 10개 단어 발음 → 학부모 3명 청취 평가
2. **복습 가치 언어 실험** (1주): "복습" vs "기억 테스트" 메시지 어느 것이 더 탭률 높은지 → TestFlight
3. **Streak 메시지 노출 위치** (1주): 홈 화면 상단 vs 게임 결과 화면 어느 것이 효과적인지
